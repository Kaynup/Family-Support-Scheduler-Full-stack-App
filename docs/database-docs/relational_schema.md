## Purpose Of The Table

This table stores a single bill record per row. The schema is shaped around the business idea of one bill having a title, a monetary amount, two dates, a status, and an optional grouping label.

## Table Identity

- Primary key: auto-increment integer id.
- Identity rule: the id is the stable way to reference a bill in update/delete operations.

## Column Commentary

### `id`

- Type: integer.
- Behavior: auto-generated by the database.
- Why it exists: every bill needs a unique row identity.

### `name`

- Type: text / string.
- Meaning: the bill label the user sees.
- Why it exists: it gives the bill a human-readable title.

### `creation_date`

- Type: date.
- Meaning: the date the bill was created in the application.
- Why it exists: it helps track when the record entered the system.

### `due_date`

- Type: date.
- Meaning: the date on which the bill becomes due.
- Why it exists: the upcoming-bills logic depends on it.

### `total_amount`

- Type: numeric / float.
- Meaning: the amount attached to the bill.
- Why it exists: the bill would be incomplete without a value to pay.

### `status`

- Type: text / string.
- Meaning: whether the bill is currently paid or unpaid.
- Why it exists: this drives status updates and filtering.

### `category`

- Type: text / string, optional.
- Meaning: a user-defined grouping label.
- Why it exists: bills can be grouped by purpose without making that grouping mandatory.

## Field Relationship Notes

- `creation_date` and `due_date` are both dates but serve different purposes.
- `status` is not used as a unique identifier; `id` is.
- `category` can be empty without breaking the record.
- `due_date` supports the query that finds bills due within a number of days.

## Design Intent

The schema is intentionally small and direct because the app is focused on bill tracking rather than complex accounting.

That means:

- the table is easy to query,
- the update/delete path is id-based,
- validation happens before write operations,
- and the schema stays readable enough to inspect without ORM abstraction.
